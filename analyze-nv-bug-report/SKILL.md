# Analyze NVIDIA Bug Report Logs

## Overview

Parse log files generated by `nvidia-bug-report.sh`, extract key GPU diagnostic information and output a human-readable report.

**Skill install paths** — this skill lives at one of:

- Cursor: `~/.cursor/skills/analyze-nv-bug-report/`
- Claude Code: `~/.claude/skills/analyze-nv-bug-report/` (user-level) or `.claude/skills/analyze-nv-bug-report/` (project-level)
- Codex: `~/.codex/skills/analyze-nv-bug-report/`

All `scripts/...` paths below are **relative to this skill's root**. When generating a shell command, prepend the install root for the active tool (e.g. on Claude Code: `python3 ~/.claude/skills/analyze-nv-bug-report/scripts/analyze.py ...`). `SKILL.md` and `scripts/` live at the root; bundled tools are under `scripts/third_party/`.

## Workflow

### Step 1: Prepare Files

If the file is in `.gz` compressed format, decompress first:

```bash
gunzip -k <file>.log.gz
```

### Step 2: Extract and Analyze Key Information

Use the bundled analysis script for automated extraction. The script automatically:
- Extracts system info, lspci, nvidia-smi, dmesg GPU diagnostic data
- Falls back to `/proc/driver/nvidia/gpus/*/information` and dmesg serial numbers when `nvidia-smi --query` fails (e.g. dead GPU causes the entire command to error out)
- Supplements Driver Version, CUDA Version, and OS from `nvidia-smi --query --unit`, `/etc/issue`, and dmesg when not found in the log header
- Extracts Xid errors from ALL available log sections: `Scanning kernel log files` (including `/var/log/kern.log` with previous-boot history), `journalctl -b -0:`, `/var/log/messages`, and standalone `dmesg:` — all sources are mandatory and scanned independently with cross-section deduplication
- Detects Xid errors and invokes `third_party/nvidia_xid_analyzer.py` for deep decoding (mnemonic, severity, HW/SW classification, resolution, etc.)
- Detects derivative Xid entries via two mechanisms: (a) NVRM lines explicitly tagged `caused by previous Xid N`, and (b) **time-inferred**: bare Xid 45 lines emitted within 10s of a primary Xid 145/149 on the same BDF (NVRM driver intermittently drops the textual annotation on later cleanup channels in the same burst). Each xid dict gets `is_derivative` / `caused_by`, with an extra `derivative_inferred=True` flag for the inferred variant so 7.1 can call out the inferred count. Section 7.1 splits Primary vs Derivative counts (and footnotes the inferred subtotal); section 7.2 decodes **every** Xid type (primary and derivative both feed the analyzer so the table reflects all observed mnemonics; downstream `(decoded_xid, mnemonic, resolution)` dedup keeps it compact regardless of how many derivative lines were processed); section 7.3 collapses each `(xid, caused_by)` derivative group to a single representative line per burst with a `# +N more ... suppressed` annotation, so a single Xid 145 + dozens of Xid 45 channel cleanups no longer floods the report
- Parses `/var/log/nvidia-imex.log` for "Node disconnect event detected" ERROR entries only (WARNING and other ERROR types are filtered out), groups by timeline events, and intelligently deduplicates repetitive messages
- Correlates IMEX disconnect events with Xid bursts by time overlap (+/- 60s) for causal analysis, showing related IMEX **event numbers** in each Xid Raw Log event and **interleaving IMEX log lines into the raw log block**
- Generates a complete Markdown report and saves it to the log directory

```bash
python3 scripts/analyze.py <file>.log
```

Optional arguments:
- `--output-dir DIR`: Specify report output directory (defaults to `report/` subdirectory under the log file location)

Report file is named `<original-filename>-analysis-report.md`, saved in the `report/` directory.

**HTML sidecar output:** alongside every `.md` report (single-file and cross-node), a self-contained `.html` sidecar with the same basename is written to the same directory. The HTML version is the recommended viewer for wide tables (NVLink §5 with 4×18 matrices, the cross-node Xid comparison matrix) — it provides:
- Sticky table header and sticky first column (no horizontal scroll loses context).
- Non-zero numeric cells highlighted; zeros and placeholders muted.
- Auto-generated left sidebar TOC with active-section highlight.
- Click-header table sort, top-bar live row filter, expand/collapse all `<details>`.
- Auto light/dark theme via `prefers-color-scheme`.
- Fully self-contained: CSS+JS embedded, no CDN, opens offline in any browser.

Open with `open <name>.html` (macOS) / `xdg-open <name>.html` (Linux).

The HTML sidecar requires the `markdown` PyPI package (see Step 4 dependency list). If `markdown` is missing or HTML rendering fails for any reason, a `[warn]` line is printed to stderr and the `.md` report is still written normally — HTML failure never blocks the markdown output.

**Platform support:** Linux and macOS. Batch mode runs each log in its own process (Linux uses `fork`, macOS uses `spawn` — same throughput, slightly slower per-worker cold start on macOS).

#### Batch Analysis Mode

Analyze multiple nv-bug-report files and generate a cross-node Xid comparison report:

```bash
# Batch analyze multiple files
python3 scripts/analyze.py --batch file1.log.gz file2.log.gz file3.log.gz --output-dir DIR

# Batch analyze all nv-bug-reports in a directory
python3 scripts/analyze.py --batch /path/to/logs/*nv-bug-report*.log.gz --output-dir DIR
```

Batch mode will:
1. Analyze each file individually, generating its own analysis report
2. Generate a `cross-node-report.md` comparison report containing:
   - File overview (hostname, system SN, chassis SN, slot#, tray#, boot time, message start time, collect date)
   - IMEX Node Disconnect Timeline (all nodes' "Node disconnect event detected" ERROR entries merged and sorted by time, grouped by burst, gap > 1 min = new event)
   - Xid comparison matrix (node × Xid number) + companion subsection 3.1 NVLink Sub-type Breakdown (per host × GPU × Xid 144-150 × Category × Severity, long format; collapsed by default; entire subsection omitted when no Xid 144-150 anywhere in the batch)
   - Xid unified timeline (all files' Xid merged and sorted by time, grouped by burst, gap > 1 min = new event; each event annotated with related IMEX disconnect events within +/- 1 min)
   - Xid decode summary (merged and deduplicated)
   - Cross-node comparison summary (common/unique Xid types analysis; agent appends dmesg/messages cross-node Analysis Summary as subsection 6.1 per Step 2.6)

If the script is unavailable or fails, follow the manual analysis flow below.

### Step 2.5: NVBugs Xid Association Search (Optional, disabled by default)

**By default, skip this step entirely. Do NOT call any NVBugs MCP tool, do NOT append any NVBugs section to the report, and do NOT add any fallback note unless the user explicitly asks for NVBugs lookup.**

Run this step only when the user's request explicitly opts in to NVBugs lookup (natural-language signals: "search nvbugs", "look up related NVBugs", "include NVBugs related bugs", "查 nvbug", "搜索 nvbugs", "查相关 bug", or any equivalent phrasing). When the user does not explicitly opt in, silently skip this step and continue with the normal report workflow.

**When the user explicitly requests NVBugs lookup, still skip this step entirely (do NOT append any NVBugs section to the report; do NOT add any fallback note) if ANY of the following holds:**

1. **MCP unavailable** — on the first `nvbugs_search` call attempt:
   - server `user-MaaS NVBugs` is not registered, OR
   - the call returns a connection / timeout / authentication error, OR
   - the call returns an unrecoverable error (not a "no results" empty response — empty results are still a successful call and are simply rendered as an empty subsection)

   When this happens, abort the entire NVBugs subsection (do not retry per Xid type), and tell the user once in chat ("NVBugs MCP unavailable, skipping NVBugs search.").

2. **No Xid errors** — section 7.2 (Xid Detailed Decode) of the per-node report (or its equivalent in the cross-node report) contains zero decoded Xid entries. Silently skip; do not mention in chat.

Otherwise, after report generation use MaaS NVBugs MCP to search related bugs and append to the report.

**Procedure:**

1. Extract the deduplicated Xid type list (Xid number + mnemonic) from report section 7.2 (Xid Detailed Decode)
2. Get the GPU model (e.g. GB200) from report section 2
3. For each unique Xid type, call the `nvbugs_search` MCP tool for semantic search:
   - server: `user-MaaS NVBugs`
   - toolName: `nvbugs_search`
   - arguments: `{ "query": "GPU Xid {number} {mnemonic} on {gpu_model}", "search_type": "semantic", "max_results": 5 }`
4. Skip Xid 45 (preemptive cleanup, typically triggered by other Xid, not an independent issue)
5. Extract bug ID, title, severity, disposition from search results
6. Append results to the end of the report file as the next sequential section (e.g. "9. NVBugs Related Bugs" for single-node reports, "7. NVBugs Related Bugs" for comparison reports — use the next number after the last existing section)

**Example search queries:**
- `"GPU Xid 145 NVLINK_RLW_ERROR on GB200"`
- `"GPU Xid 79 GPU fallen off the bus"`
- `"GPU Xid 119 GSP_RPC_TIMEOUT"`

### Step 2.6: dmesg / messages Analysis

`analyze.py` automatically scans both the `dmesg:` section and syslog/messages sections (`journalctl`, `/var/log/messages`), deduplicates across sections, and embeds non-GPU warnings as **Section 7.5 Other Warnings** (after 7.4 Other GPU Related). GPU/NVIDIA/PCIe related lines are excluded from 7.5 — they are already covered by 7.3 (Xid Raw Logs with interleaved NVRM context) and 7.4 (Other GPU Related).

The embedded highlights have two categories:
- **System Errors** — kernel panic, oops, OOM, MCE, hardware errors, etc.
- **Other Warnings** — timeouts, temperature, network, storage warnings, etc.

**Agent Procedure (Comparison Report Only):**

For single-node reports, section 7.5 is auto-generated — no agent action needed.

For comparison reports (batch mode), append the cross-node dmesg/messages analysis **into Section 6 (Cross-Node Comparison Summary)** as subsections, directly after the auto-generated Xid comparison content (common/unique Xid, IMEX-Xid correlation). Do NOT create a separate top-level section.

1. Read each single-node report's section 7.5 highlights
2. Append a single `### 6.1 Analysis Summary` subsection under section 6. It MUST start with an LLM-generated disclaimer note and wrap its content in a collapsible `<details>` block. Group findings by theme (omit a theme if no relevant data) — do NOT split into separate "Issue Summary" and "System-Level Analysis" subsections:

```markdown
### 6.1 Analysis Summary

> **Note**: This subsection is generated by the LLM agent by reading each node's section 7.5. For reference only.

<details>
<summary>Click to expand Analysis Summary</summary>

Group findings by theme. Recommended themes (omit a theme if no relevant data):

- **Xid Overview**: how many nodes have Xid errors, dominant Xid types, nodes with no Xid
- **Most Affected Node(s)**: which node has highest Xid count or unique critical Xid types, with escalation chain
- **Multi-Node / Fabric-Level Failure**: simultaneous Xid bursts across nodes, same Chassis SN, fabric-level inference
- **IMEX <-> Xid Correlation**: which IMEX events overlap which Xid events; IMEX import failures across nodes
- **Causal Chain**: dominant Xid causal sequence (e.g. Xid 145 -> Xid 45) and any escalated chain on specific nodes
- **System-Level Health**: kernel panic / oops / OOM / MCE / hardware errors (state "none" if absent); non-GPU misc (disk reset, fuse taint, overlayfs, etc.) with brief impact note
- **Operational Traces**: any manual troubleshooting evidence (e.g. nvidia-smi -r, kill -9 GPU procs, container toolkit reinstall) found in section 7.5

</details>
```

Markdown formatting requirements for the collapsible blocks:
- No blank line between `<details>` and `<summary>`
- Keep one blank line after `</summary>` before markdown content
- Keep one blank line before `</details>`

Do NOT generate "Root Cause Hypothesis" or "Recommended Actions" subsections.

**Notes:**
- If more context is needed (e.g. 10 lines around an error), use the Read tool to directly read the original log file
- If a node's 7.5 section is empty or missing, skip that node

### Step 3: Manual Analysis Guide

#### 3.1 Basic Information

Extract from file header (first 70 lines):
- Date, hostname, kernel version, OS distribution
- NVIDIA driver version, CUDA version
- GPU count and model summary

#### 3.2 lspci Analysis

Three lspci sections exist in the file:

| Section | Focus |
|---------|-------|
| `/sbin/lspci -nntv` | PCI topology tree, quick GPU count and placement check |
| `/sbin/lspci -nn` | Identify all NVIDIA 3D controllers (`[0302]`), note BDF addresses |
| `/sbin/lspci -nnDvvvxxxx` | **Key**: check the following fields for each GPU device |

For each GPU (search `3D controller.*NVIDIA`), check:

1. **PCIe Link Status**: compare `LnkCap` and `LnkSta`
   - Normal: `LnkSta: Speed XXG/Ts (ok), Width xN (ok)`
   - Abnormal: `(downgraded)` indicates link degradation
   - Check whether Speed and Width reach LnkCap expected values

2. **Device Status** (`DevSta`):
   - `CorrErr+` = correctable error
   - `NonFatalErr+` = non-fatal error
   - `FatalErr+` = **fatal error (critical)**
   - `UnsupReq+` = unsupported request

3. **AER Errors**:
   - `UESta:` uncorrectable error status (items with `+` are abnormal)
   - `CESta:` correctable error status
   - Watch for: `DLP+`, `SDES+`, `FCP+`, `CmpltTO+`, `MalfTLP+`

4. **PCIe BAR Region**: check BAR memory mappings
   - Region numbers vary by GPU model (e.g. Region 0/2/4 for GB200, Region 0/1/3 for H20) — they are auto-detected from lspci data
   - Abnormal: Region marked `[disabled]` or `[virtual]` (no physical address assigned)
   - Abnormal: Region sizes differ between same-model GPUs

#### 3.3 nvidia-smi Analysis

Search `/usr/bin/nvidia-smi --query`, for each GPU (starting with `GPU 0000xxxx:xx:xx.x`) extract:

| Field | Search Keyword | Anomaly Criteria |
|-------|---------------|-----------------|
| Name/SN/UUID | `Product Name`, `Serial Number`, `GPU UUID` | Identification |
| PCIe Link | `Link Width Current` | Mismatch with Max is abnormal |
| ECC Errors | `ECC Errors` section | Uncorrectable > 0 is abnormal |
| Retired Pages | `Retired Pages` | Any retired pages need attention |
| Remapped Rows | `Remapped Rows` | Uncorrectable > 0 or Pending=Yes is abnormal |
| Temperature | `GPU Current Temp` | Large variance (>10°C) between GPUs needs attention |
| Power | `Power Draw` | Large variance between GPUs needs attention |
| Clock Events | `HW Slowdown`, `HW Thermal Slowdown` | Active is abnormal |
| NVLink | Search `nvidia-smi nvlink --errorcounters` | error count > 0 needs attention |

#### 3.4 dmesg / messages Analysis

Search ALL of the following sections for Xid and NVRM errors:

1. **`Scanning kernel log files for NVIDIA kernel messages:`** — MUST be scanned; contains `/var/log/kern.log` which may hold Xid from **previous boots** not present in current-boot journalctl or dmesg
2. **`journalctl -b -0:`** — current boot syslog
3. **`/var/log/messages`** — alternative syslog (present on some systems)
4. **`dmesg:`** — current boot kernel ring buffer

> **Important**: All four sources MUST be scanned independently. Do NOT skip any section even if another section is found — kern.log often contains historical Xid from prior boots that journalctl -b -0 does not include.

Within these sections, search for:

1. **Xid Errors**: search `NVRM.*Xid` or `Xid.*nvidia`
   - Record each Xid's timestamp, GPU BDF, Xid number
   - Common critical Xids: 13(GR), 31(GPU memory page fault), 43(GPU stopped), 45(preemptive cleanup), 48(DBE), 61/62/63(internal firmware), 64(ECC page retirement/row remap), 69(graphics context), 74(NVLink), 79(GPU fallen off bus), 92/94/95(high single/double-bit ECC)
   
2. **GPU Fallen Off**: search `fell off the bus`, `GPU has fallen off`
3. **NVRM Errors**: search `NVRM:` lines excluding `loading` and `Persistence`
4. **PCIe AER**: search `AER`, `PCIe Bus Error`

### Step 4: Supplemental Xid Deep Decode (Manual Flow)

If Step 2's script failed to automatically invoke nvidia_xid_analyzer, decode manually:

```bash

```

Note: The same Xid number may produce different classifications and resolutions depending on subcode/intrinfo/errorstatus; preserve all decode variants.

## Notes

- nv-bug-report logs are typically very large (tens of thousands to millions of lines); use Grep to locate key sections, avoid reading entire file
- Sections are separated by `____________________________________________` (a line of underscores)
- GPU devices in lspci are identified as `3D controller [0302]: NVIDIA`
- nvidia-smi query section marks each GPU's info block with `GPU 0000xxxx:xx:xx.x`
- dmesg section starts with `dmesg:`
- If there are multiple GPUs, **comparing each GPU's status individually** is key to finding the anomalous GPU
